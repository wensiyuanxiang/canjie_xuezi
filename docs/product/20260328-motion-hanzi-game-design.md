# 仓颉学字 - 体感切字版详细方案

日期：2026-03-28
目的：将“运动中学习汉字”的方向收敛为可执行的手机网页版本方案，明确主题、界面、布局与技术路线。

---

## 一、产品定位

### 1.1 一句话定义

孩子举起手机，用前置摄像头识别手指，在镜头前划动“切开”目标汉字，在运动中完成识字，并逐步修复一个失落的文字世界。

### 1.2 产品关键词

- 体感识字
- 划切爽感
- 世界修复
- 商业游戏包装
- 手机网页可玩

### 1.3 产品目标

这不是“套了 UI 的识字题库”，而是一款有明确主线、有主题包装、有官卡推进、有进度追踪的轻度商业化教育游戏。

本方案以黑客松 MVP 为目标，但界面和结构按“商业游戏感”设计。

---

## 二、核心玩法方向

### 2.1 核心动作

玩家只做一件事：

- 把食指伸到前置摄像头前
- 控制屏幕中的“金色笔锋指针”
- 划过目标汉字，完成“切字”

不采用多手势，不做复杂手型分类。

当前版本只识别：

- 一根食指的位置
- 食指轨迹速度
- 是否与目标字发生命中

### 2.2 核心循环

单局循环如下：

1. 关卡开始
2. 屏幕提示本关目标
3. 汉字批量进入场景
4. 玩家用食指划切目标字
5. 正确命中触发特效、收集、连击
6. 关卡结束后展示学习结果
7. 世界地图进度推进，解锁下一关

### 2.3 为什么“切字”比“接字”更适合体感

- 手指在镜头前做划动，比做精确接住更自然
- 划切动作更容易产生“命中爽感”
- 轨迹类交互更适合做笔锋、墨迹、裂开、飞散等视觉反馈
- 和汉字书写、笔画动作有天然联想

### 2.4 学习闭环

每一关都必须形成最短学习闭环：

- 看见字形
- 听见读音
- 知道一个词或一个意象
- 在结算中再次强化

游戏进行中以“认形”为主，结算页补“音”和“义”。

## 三、MVP 范围

### 3.1 本轮必须做

- 首页
- 设置页
- 世界地图
- 山野主题 6 个官卡
- 前置摄像头校准页
- 关卡内切字玩法
- 结算页
- 已学汉字进度展示
- 解锁和存档

### 3.2 本轮不做

- 三纪完整内容
- 八大世界全量
- 开放式 AI 互动
- 多人模式
- AR 空间定位
- 多指复杂手势
- 实时骨骼特效大系统

### 3.3 MVP 展示目标

评委或玩家在 20 秒内能看懂：

- 这是一个用手指切字的游戏
- 它不是普通答题，是运动中识字
- 它有完整世界、关卡、进度和解锁

## 四、世界观与主题包装

### 4.1 核心主题

主题延续 GDD：

- 世界曾由文字构成
- 文字失落后，山川、河流、草木、村落都变得模糊
- 孩子作为“接字人/执笔者”，重新收回文字，世界逐渐复苏

### 4.2 本轮主题世界

MVP 只做第一世界：

- 世界名称：山野
- 主题字：山、水、木、火、土、石
- 主视觉基调：灰白山水逐渐恢复为彩色山野

### 4.3 主进度表达

玩家需要看到一个明确主体进度。

本方案采用双进度：

- 世界修复进度：`山野修复 0% -> 100%`
- 字库收集进度：`已学 0/6`

这两个进度同时存在：

- 世界修复带来“做游戏”的感觉
- 字库收集带来“在学习”的感觉

### 4.4 关卡叙事

山野 6 关结构建议：

1. `山`：世界出现山形
2. `水`：河流显现
3. `木`：树木生长
4. `火`：暖色光点与篝火
5. `土`：地面和坡道恢复
6. `石` / BOSS：巨石字灵被解放，世界完整恢复

每关结束时，不只显示“你过关了”，而是显示“你修复了什么”。

## 五、视觉风格方案

### 5.1 风格关键词

- 国风童话
- 治愈感
- 水墨与金光结合
- 轻商业手游感
- 干净，不幼稚

### 5.2 风格方向

不是纯传统国风，也不是廉价儿童早教风。

目标是：

- 有东方文字感
- 有游戏包装感
- 有儿童亲和力
- 不做廉价拼贴

视觉组合建议：

- 背景：低对比水墨山水
- 交互主元素：金色书法笔锋、暖橙光晕
- UI 面板：纸张、卷轴、木牌、云纹
- 动效：墨迹、金粉、裂光、飞字

### 5.3 色彩方案

基础变量：

```css
--color-bg: #f5f0e8;
--color-ink: #2d3436;
--color-primary: #ff9f43;
--color-secondary: #6c8ebf;
--color-accent: #ffd93d;
--color-success: #6bcb77;
--color-danger-soft: #d97d7d;
--color-panel: rgba(255,255,255,0.92);
--color-mist: #dfe6e9;
```

使用原则：

- 目标字：暖橙 + 金光
- 干扰字：灰蓝 + 冷边
- 背景：米白、灰绿、雾灰
- 重要状态：金色
- 学习成功：柔绿色

### 5.4 字体建议

- 汉字主字体：`Noto Sans SC Bold` 或 `Source Han Sans Bold`
- 数字与 UI 字体：同套字体，避免风格撕裂
- 标题可以额外做一版图片 Logo，不建议混用太多字体

### 5.5 动效语言

统一动效语义：

- 划中目标字：裂开、停顿 80ms、金光喷发、飞入收集槽
- 划中错误字：轻微晃动、冷色碎点、字灵提示
- 连击：屏幕边缘金光呼吸
- 通关：背景从灰到彩缓慢溶解
- 解锁：地图节点发亮，云雾散开

### 5.6 商业游戏感来源

商业感不来自写实，而来自一致性和层级。

需要统一：

- 按钮形状
- 面板材质
- 图标语言
- 过场节奏
- 反馈强弱

任何页面都不允许临时拼风格。

## 六、页面结构设计

### 6.1 页面总览

MVP 页面包括：

- 首页 `Home`
- 设置页 `Settings`
- 校准页 `Calibration`
- 世界地图 `WorldMap`
- 关卡页 `Game`
- 结算页 `Result`
- 图鉴页 `Collection`

### 6.2 首页设计

目标：

- 第一眼就让用户知道这是“国风 + 汉字 + 体感”的游戏

布局：

- 顶部：Logo「仓颉学字」
- 中部：半灰半彩的山野主视觉
- 中下：字灵角色 + 一句简短文案
- 底部主按钮：`开始修复`
- 次按钮：`图鉴`、`设置`

背景表现：

- 画面中心有一本发光古书或金色字雨
- 远景是尚未修复的山野世界

首页文案建议：

- 主文案：`举起手指，让文字归来`
- 副文案：`在运动中学汉字，在游戏里修复世界`

### 6.3 设置页设计

设置页不是附属页面，而是商业感的重要组成部分。

必须有：

- 音乐开关
- 音效开关
- 左右手模式
- 摄像头预览开关
- 识别灵敏度
- 重置校准

建议布局：

- 顶部标题栏
- 中部卡片式设置项
- 底部返回按钮

额外项：

- `低性能模式`
- `开启辅助高亮`

### 6.4 校准页设计

这是摄像头玩法成败关键页面。

目标：

- 用 5 到 10 秒让用户理解如何把手指映射到屏幕中

布局：

- 顶部：标题 `校准手指`
- 中央：前置摄像头预览区域
- 中间覆盖一个圆形目标框
- 底部：说明文字和下一步按钮

流程：

1. 提示允许摄像头权限
2. 提示举起一根食指
3. 检测到指尖后显示金色圆点
4. 引导用户把指尖移到指定区域
5. 完成后进入地图或关卡

校准反馈：

- 检测成功：绿色勾线和轻粒子
- 未检测到：灰色提示，不报错惊吓

### 6.5 世界地图设计

这是商业游戏感最重要的页面之一。

地图目标：

- 让玩家看到“章节推进”
- 让玩家理解“我在修复世界”

布局建议：

- 顶部：世界标题 `山野`
- 顶部右侧：设置入口
- 顶部下方：世界修复进度条 + 已学字数
- 中部：纵向卷轴式地图
- 地图上有 6 个关卡节点
- 底部：返回首页 / 图鉴入口

地图视觉：

- 起始是灰白地图
- 已解锁区域出现淡彩
- 已通关节点发金色光
- 当前关卡有脉冲呼吸效果

地图节点类型：

- 普通关：圆形印章
- BOSS 关：较大金红印章
- 未解锁：灰白锁

### 6.6 关卡页设计

关卡页必须把“摄像头玩法”和“汉字辨认”同时做好。

布局采用三段式：

#### 顶部 HUD

- 左上：返回
- 中上：本关目标字/任务说明
- 右上：连击数、暂停

#### 中部战斗区

- 背景：山野场景
- 汉字在中上区域生成与移动
- 指尖光标和笔锋轨迹在场景上层

#### 底部状态区

- 当前命中目标数
- 本关收集槽
- 字灵提示区

关卡核心视觉点：

- 摄像头预览不作为主画面铺满，只做小窗或完全隐藏
- 玩家感知的是“我的手指变成笔锋”，不是在看自拍

### 6.7 结算页设计

目标：

- 强化学习闭环
- 给出成长反馈
- 激励继续下一关

布局：

- 顶部：通关评价
- 中部：本关学到的字卡
- 中部下方：读音按钮 / AI 小故事
- 下方：世界修复变化预览
- 底部按钮：`下一关`、`返回地图`

重点信息：

- 接中率
- 连击纪录
- 新学汉字
- 修复了什么元素

### 6.8 图鉴页设计

图鉴页是“学习成果可视化”的核心。

布局：

- 顶部：总字数与完成度
- 中部：按世界分类的汉字卡片
- 点击字卡可查看：
  - 大字
  - 拼音
  - 简短词语
  - 是否掌握

MVP 中图鉴不用太复杂，但必须存在。

## 七、关卡玩法细化

### 7.1 基础模式：切中目标字

规则：

- 关卡开始显示目标字，例如 `切开“山”`
- 场景中出现目标字和干扰字
- 食指轨迹划过目标字判定成功

成功反馈：

- 字裂开
- 金色碎片
- 音效
- 计入连击

失败反馈：

- 划过错误字时轻微冷色碎点
- 连击中断
- 不重罚

### 7.2 二级变化模式

MVP 可以在后两关加 1 到 2 个轻变化：

- 听音切字：只播读音，不直接显示目标大字
- 顺序切字：先切 `山` 再切 `水`
- 偏旁切字：切所有带 `木` 的字

原则：

- 不改核心交互，只改识字目标
- 保持“切字”始终是唯一输入

### 7.3 BOSS 关

建议 BOSS 不是打怪，而是“解封大字”。

例如 `岩` BOSS：

- 屏幕中心有灰白大字 `岩`
- 周围飞出部件字 `山`、`石`
- 玩家按顺序切中组件
- 全部命中后中央大字被点亮并解放

这样既保留游戏高潮，也强化汉字构形。

## 八、手感设计原则

### 8.1 爽感来源

爽感主要来自以下组合：

- 低延迟指针
- 清晰命中判定
- 强反馈停顿
- 夸张收集动画
- 明确连击增长

### 8.2 指尖表现

屏幕中的手指不要直接显示真实手指模型，建议表现为：

- 一个金色光点
- 后面拖一条半透明笔锋
- 高速划动时拖尾加粗

这会更像“执笔切字”，也更容易掩盖识别抖动。

### 8.3 命中判定

命中判定宁宽不严。

建议：

- 字的视觉区域外扩一个隐藏命中半径
- 高速划过时做轨迹线段碰撞，而不是只看单帧点坐标
- 对儿童玩家容错更高

### 8.4 节奏控制

每局 45 到 60 秒

每 8 到 12 秒给一次小高潮：

- 更密集的一波目标字
- 明显连击机会
- 一次场景反馈

不能让节奏全程平。

## 九、技术路线

### 9.1 前端架构

建议技术栈：

- `React 18 + TypeScript`
- `Vite`
- `Phaser 3`
- `MediaPipe Tasks Vision` 或浏览器可用的 Hand Landmarker
- `Zustand` 或轻量全局状态方案
- `localStorage` 做本地存档

职责划分：

- React：页面、导航、设置、地图、图鉴、结算
- Phaser：关卡内实时玩法、字对象、特效、碰撞
- Hand Tracking：独立输入层，不直接耦合 UI

### 9.2 输入识别路线

推荐路线：

1. 浏览器申请前置摄像头权限
2. 将视频流送入手部关键点模型
3. 取食指指尖关键点坐标
4. 映射到游戏坐标系
5. 生成指针与轨迹
6. 交给 Phaser 做命中检测

MVP 只取一个坐标：

- `index_finger_tip`

不做的事：

- 不做复杂姿态识别
- 不做双手
- 不做 3D 手势语义分类

### 9.3 识别库建议

首选：

- MediaPipe Hand Landmarker Web

原因：

- 浏览器端可跑
- 移动端网页兼容性相对成熟
- 有现成关键点输出
- 足够支撑单指追踪

注意：

- 识别频率不必满帧
- 可按 `15-24fps` 识别，再插值到游戏帧

### 9.4 坐标映射方案

关键问题不是“识别到手”，而是“映射稳定”。

建议流程：

- 视频预览镜像
- 游戏坐标单独做归一化映射
- 左右手模式切换时只切映射规则，不动玩法逻辑
- 增加轻量平滑滤波，降低抖动

建议处理：

- 使用滑动平均或指数平滑
- 超出边界时做钳制
- 突发跳点时丢弃异常值

### 9.5 游戏层实现

Phaser 中建议拆成以下模块：

- `InputTracker`
  - 接收手指坐标
  - 维护轨迹点
- `SlashTrail`
  - 负责笔锋和拖尾显示
- `FallingChar`
  - 单个字对象
- `HitSystem`
  - 轨迹和汉字的碰撞判定
- `ComboSystem`
  - 连击与特效节奏
- `LevelManager`
  - 当前关卡配置和胜负判断

### 9.6 摄像头预览策略

不建议把自拍画面作为主舞台背景。

建议两种方案二选一：

- A 方案：关卡页隐藏摄像头画面，只显示校准页预览
- B 方案：关卡页右下角小窗显示摄像头预览

MVP 推荐 A 方案。

原因：

- 更像游戏，不像工具
- 减少视觉干扰
- 性能更稳

### 9.7 状态管理

建议全局存以下内容：

- 当前页面
- 摄像头权限状态
- 校准状态
- 设置项
- 已通关关卡
- 已学汉字
- 当前世界进度

本地存档建议字段：

- `unlockedLevels`
- `completedLevels`
- `learnedChars`
- `bestCombos`
- `settings`
- `calibrationProfile`

### 9.8 性能策略

手机网页要优先稳，不优先炫。

规则：

- 手部识别和渲染解耦
- 粒子数量控制
- 汉字对象池复用
- 动画尽量使用简单 tween
- 背景层级控制在 2 到 3 层

必要时提供：

- `低性能模式`
- 降低粒子
- 降低轨迹采样点
- 关闭摄像头小窗

## 十、页面布局尺寸建议

### 10.1 基准画布

建议以手机竖屏为主：

- 设计基准：`390 x 844`
- 游戏逻辑基准：可按 `1080 x 1920` 比例设计后缩放

### 10.2 关卡页安全区

建议分区：

- 顶部 HUD：约 14%
- 中部主战区：约 66%
- 底部状态区：约 20%

汉字主要活动范围：

- 屏幕高度 `18% - 78%`

底部不要堆过多按钮，避免挡住体感操作注意力。

## 十一、音频设计

### 11.1 BGM

建议：

- 首页：梦幻、温柔
- 地图：探索感
- 关卡：轻快、有节奏
- 结算：短促喜悦

### 11.2 SFX

必须有：

- 正确切中
- 错误切中
- 连击升级
- 通关
- 解锁
- 页面点击

切中音效要非常关键，应该像“切开一张发光纸片”，不是像砍杀。

## 十二、MVP 研发顺序

建议严格按顺序做：

1. 跑通摄像头权限与食指坐标读取
2. 在网页里显示稳定的指尖光标
3. 在 Phaser 中实现划线和命中检测
4. 做一个只有目标字的 30 秒原型
5. 加入干扰字和连击
6. 加入结算页与地图推进
7. 再补设置、图鉴、主题美化

顺序不能反。

如果先堆页面和世界观，再回头发现指尖识别不稳，整个方案都会失真。

---

## 十三、风险与规避

### 13.1 最大风险

- 手部识别抖动
- 手机浏览器权限问题
- 低端设备掉帧
- 玩家乱划而不认字

### 13.2 规避方案

- 只识别单根食指
- 只做单核心动作
- 命中判定放宽
- 识别失败可回退到触屏模式
- 新字关降低速度和干扰密度

### 13.3 重要备用方案

必须保留触屏模式作为降级方案：

- 摄像头不可用时，直接用手指触屏划切
- 所有玩法逻辑不变，只替换输入源

这条非常重要。

它决定项目不会因为单点技术问题直接不可演示。

---

## 十四、最终产品气质

这款游戏最终要呈现的不是“一个识字工具”，而是：

- 一个可以上手玩的体感汉字冒险游戏
- 一个孩子会因为“切字很爽”而愿意反复玩的学习产品
- 一个有明确主题、包装、进度和收集目标的商业游戏雏形

一句话总结：

不是让孩子“被迫学字”，而是让孩子为了“修复世界”而主动反复识字。
